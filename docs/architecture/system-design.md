# 🏗️ System Design 

---

## 🔹Overview

This system is designed to support **cognitive focus self-regulation** during work or study sessions through behavioral data analysis.

The system:

- Captures user interaction events (activity, pauses, presence)
- Processes temporal metrics
- Generates qualitative focus indicators
- Provides **AI-generated feedback** and visualization to the user

The architecture follows a **client–server model** with **modular separation of responsibilities**, allowing scalability and future evolution.

> The system prioritizes **privacy by design**, performing visual processing locally in the client and avoiding storage of sensitive data.

---

## 🔹Core Principles

- **Separation of concerns**
- **Modular architecture**
- **Client-side processing for sensitive data**
- **Deterministic rule-based analysis (no ML initially)**
- **AI-assisted interpretation layer (non-deterministic)**
- **Scalability and extensibility**

---

## 🔹High-Level Architecture

The system is structured into 3 main layers:

### 1. Presentation Layer (Frontend)

- User interaction
- Event capture
- Visualization of results
- Local visual processing (MediaPipe)

---

### 2. Application Layer (Backend)

- API REST
- Event processing
- Metrics calculation
- Rule-based classification
- Indicator generation
- **AI-based feedback generation (interpretation layer)**

---

### 3. Data Layer (Database)

- Persistent storage of:
  - Users
  - Sessions
  - Events
  - Metrics
  - Indicators
  - Feedback (optional persistence)

---

## 🔹System Flow

The system follows this logical pipeline:

1. User interacts with the interface
2. Events are captured with timestamps
3. Events are stored in the backend
4. Metrics are calculated
5. Rules are applied
6. Indicators are generated
7. **AI generates feedback based on computed results**
8. Results are displayed to the user

📌 This flow aligns with the conceptual model described in the system design.

---

## 🔹Main Components

### 🔸Frontend

Responsible for:

- Capturing user interaction
- Detecting activity/inactivity
- Running visual processing locally
- Sending events to backend
- Displaying metrics, indicators, and feedback

---

### 🔸Backend API

Responsible for:

- Receiving events
- Validating data
- Storing structured data
- Running the analysis engine
- Generating indicators
- Coordinating AI feedback generation

---

### 🔸Analysis Engine

Core deterministic logic of the system:

- Calculates metrics:
  - Active time (Ta)
  - Inactive time (Ti)
  - Number of pauses (Np)
- Computes indices:
  - Continuity (IC)
  - Fragmentation (IF)
  - Stability (IGE)
- Applies rule-based classification

---

### 🔸AI Feedback Engine

Responsible for interpreting computed results and generating human-readable feedback.

- Receives:
  - Metrics
  - Indicators
- Generates:
  - Summary
  - Detailed explanation
  - Actionable recommendations

⚠️ This component does **NOT perform calculations**.  
It only interprets structured data generated by the Analysis Engine.

---

### 🔸Database

Stores:

- Users
- Sessions
- Events (timestamp-based)
- Metrics
- Indicators
- Feedback (optional)

Supports:

- Historical analysis
- Session comparison
- Longitudinal tracking

---

## 🔹Processing Model

The system transforms data through the following stages:

Events → Metrics → Indicators → AI Feedback → User Insight


Where:

- Events = raw observable data  
- Metrics = mathematical transformations (**deterministic**)  
- Indicators = qualitative classification (**rule-based**)  
- AI Feedback = natural language interpretation (**non-deterministic**)  

---

## 🔹Event-Based Thinking (Important)

Even though this is not a full event-driven architecture (like RabbitMQ),

internally the system behaves **event-based**:

- Each interaction generates an event
- Events are the source of truth
- Everything derives from events

This is key for:

- scalability
- replaying sessions
- future analytics

---

## 🔹Privacy & Data Strategy

- No raw images are stored
- Visual data is processed in-browser
- Only abstract events are sent to backend

Examples of stored data:

- "user inactive"
- "face not detected"
- "pause started"

---

## 🔹Scalability Considerations

The architecture allows:

- Scaling backend independently
- Adding workers in future
- Moving analysis engine to separate service
- Introducing event queues later (RabbitMQ / Kafka)
- **Moving AI processing to asynchronous workers**
- **Queue-based feedback generation for scalability**

---

## 🔹Future Evolution

Planned improvements:

- Real-time processing (WebSockets)
- Event streaming
- Machine learning models (optional)
- Adaptive thresholds per user
- Microservices architecture
- **Asynchronous AI processing (worker-based)**
- **Feedback caching strategies**

---

## 🔹Design Decisions

- Rule-based model instead of ML (initial simplicity)
- Client-side visual processing (privacy)
- Relational database (structured temporal data)
- Modular backend (future separation possible)
- **AI is used only for interpretation, not core logic**
- **Deterministic pipeline ensures consistency and reliability**
- **AI layer is decoupled and optional**

---

## 🔹MVP Focus

The MVP prioritizes:

- End-to-end flow working
- Accurate event tracking
- Correct metric calculation
- Clear user feedback (including AI-generated insights)
- Simple and maintainable architecture
